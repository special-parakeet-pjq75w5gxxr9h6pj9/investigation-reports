The Problem

393 orphaned GCP projects prefixed anyscale-tftst were leaking from the Terraform E2E test at backend/tests/bk_e2e/test_cloud_terraform.py. Each project cost ~$157/month, totaling ~$17k/month in wasted spend.

The leak has been ongoing since July 2023 when the GCP test was first added (commit 9cdc2ccd by Yue Zhang).

Root Cause

test_terraform_anyscale_v2_e2e_public_gcp (line 215) had no try/finally block. The flow was sequential with no error recovery:

1 L248   tf.apply() — Creates GCP project (anyscale-tftst-XXXX)
↓
2 L262   Register cloud with Anyscale
↓
3 L311   Wait for admin zone readiness
↓
4 L317   Delete cloud
↓
5 L324   tf.destroy() — Removes the GCP project

If any step between 1 and 5 failed — or the 30-minute Buildkite timeout (.buildkite/pipeline.predeploy.yaml:335) killed the process — tf.destroy() never ran and the GCP project persisted forever at ~$157/month.

The same pattern existed in the AWS test (test_terraform_anyscale_v2_e2e_public_aws, line 79), but AWS resources are cheaper and the blast radius is smaller since it doesn't create entire cloud accounts.

Why There Was No Safety Net

  • CI cleanup cron doesn't cover GCP projects. tools/kubernetes/ci/cleanup.py only cleans K8s namespaces and config maps. No GCP project cleanup existed anywhere in the codebase.
  • The anyscale-tftst prefix comes from an external Terraform module (terraform-google-anyscale-cloudfoundation-modules), not from code in this repo. The test clones that module at runtime (line 219-221).
  • Leaking since July 2023 when Yue Zhang added the GCP test (commit 9cdc2ccd). 32 orphaned project names were visible in the predeploy bootstrap Terraform state alone.

Fix & Remediation

1. Wrap tests in try/finally Critical Done

Ensures tf.destroy() runs even when the test fails mid-way (cloud registration error, admin zone timeout, etc.). Shipped in PR #38915. 

def test_terraform_anyscale_v2_e2e_public_gcp(setup_anyscale_test_env):
    # ... setup, tf.init(), tf.apply() ...

    try:
        # Register cloud
        cloud_controller.register_gcp_cloud(...)

        # Wait for admin zone
        CloudFunctionalVerificationController(
            cloud_controller.cloud_event_producer, cloud_controller.log
        )._prepare_verification(cloud_id, "GCP")

        # Delete cloud
        cloud_controller.delete_cloud(
            cloud_name=cloud_name, cloud_id="", skip_confirmation=True,
        )
    finally:
        # ALWAYS destroy terraform resources, even if test fails
        print("Destroying anyscale_v2_e2e_public gcp terraform (cleanup)...")
        return_code, stdout, stderr = tf.destroy(
            force=IsNotFlagged, auto_approve=IsFlagged
        )
        if return_code != 0:
            print(f"WARNING: Failed to destroy terraform: {stderr}")

Same pattern applied for test_terraform_anyscale_v2_e2e_public_aws.

Limitation: try/finally won't help when the 30-min Buildkite timeout sends SIGKILL — for that you need fix #3.

2. Delete the 393 orphaned projects Cleanup Done

Sumeet ran a manual cleanup script (cleanup_stale_gcp_terraform_projects.py) using the GCP service account from the gce-cloud-terraform-e2e-test secret. Dry run was executed first, then 393 projects were deleted. 

# List all anyscale-tftst projects under the test folder
gcloud projects list \
  --filter="name:anyscale-tftst AND parent.id=953296303039" \
  --format="value(projectId)"

# After review, delete each:
gcloud projects delete <project-id>

3. Periodic GCP cleanup cron Prevention

Add a cleanup job (alongside the existing K8s cleanup in tools/kubernetes/ci/cleanup.yaml) that deletes anyscale-tftst-* GCP projects older than 2 hours under folder 953296303039.

This is the only fix that handles the SIGKILL case from Buildkite timeouts. Sumeet will re-check on May 5, 2026 to see if projects are still accumulating. If so, the reaper job will be created.
