The Problem

On March 1, 2026, the roast-server-reverse-proxy pod in the roast namespace (production cluster) was killed and restarted by Kubernetes. Only one replica was affected. The pod had been running continuously since February 7 (~22 days).

The pod exited cleanly (exit code 0) after receiving a SIGTERM, but the restart interrupted Loki log ingestion via /loki/api/v1/push forwarding.

This is a recurring issue — it has happened before with no prior RCA. The same failure pattern could affect any long-running roast proxy replica.

Investigation Trail

Initial hypothesis: connection/goroutine accumulation

Pei's initial theory was that too many open connections or accumulated goroutines were blocking the /healthz endpoint. No CPU, memory, or disk pressure was observed on the node.

Logs from the restarted pod

The proxy logged a clean shutdown sequence triggered by an external signal:

{"level":"info", "caller":"reverse_proxy.go:181",
 "msg":"Forwarding request /loki/api/v1/push"}
{"level":"info", "caller":"reverse_proxy.go:88",
 "msg":"Received shutdown signal, stopping proxy server..."}
{"level":"info", "caller":"reverse_proxy.go:103",
 "msg":"Shutting down proxy server..."}
{"level":"error","caller":"reverse_proxy.go:77",
 "msg":"Error starting proxy server: accept tcp [::]:8446: use of closed network connection"}
{"level":"info", "caller":"reverse_proxy.go:118",
 "msg":"Proxy server stopped successfully"}

The shutdown was triggered via signal.NotifyContext (listening for os.Interrupt and syscall.SIGTERM). It was not a suture self-termination.

Dead end: K8s events not found by pod name

Pei searched CloudWatch audit logs filtered by pod name and found no events before the restart. This initially made the cause mysterious — it appeared the pod just received a SIGTERM from nowhere. 

aws logs filter-log-events \
    --log-group-name "/aws/eks/production_cluster/cluster" \
    --start-time 1772373573000 \
    --end-time 1772375400000 \
    --filter-pattern '{ $.objectRef.name = "roast-server-reverse-proxy-7d67b94fd7-xdhpr" }' \
    --max-items 50

Result: no events. The filter was too narrow — K8s events use objectRef.resource = "events" at the namespace level, not the pod name in objectRef.name.

Chris Sivanich queried the audit logs by namespace + resource type instead of pod name, revealing 6 consecutive liveness probe failures:

Root Cause

The Kubernetes liveness probe on http://10.0.13.174:8445/healthz timed out 6 consecutive times over ~30 seconds, triggering K8s to kill the container. The SIGTERM from K8s was then handled by the Go process via signal.NotifyContext, producing the clean shutdown logs.

Why did /healthz stop responding?

┌──────────────────────────────────────────────────────┐ │ roast-server-reverse-proxy (Go) │ │ │ │ :8446 ─── request handler ──► /loki/api/v1/push │ │ (forwarding goroutines) │ │ │ │ :8445 ─── /healthz endpoint │ │ (shares same goroutine pool / event loop) │ │ │ └──────────────────────────────────────────────────────┘ │ If request goroutines accumulate or block on upstream Loki... │ ▼ /healthz can't respond within liveness probe timeout │ ▼ K8s kills pod after 6 failures

The most likely cause is goroutine or connection saturation in the reverse proxy:

  • The proxy forwards high-volume /loki/api/v1/push requests to the upstream Loki ingester
  • If the upstream Loki becomes slow (backpressure, GC pause, disk I/O), forwarding goroutines accumulate waiting for responses
  • The /healthz handler shares the same HTTP server or goroutine pool, so it gets starved
  • After ~22 days of uptime, connection state may have accumulated (leaked goroutines, unclosed connections, or exhausted file descriptors)
Observability gap: K8s events for the GCPlane roast namespace are not exported to Loki. Debugging required manual CloudWatch audit log queries, which cost significant investigation time and required non-obvious filter patterns.

Proposed Fix

1. Separate health check listener Critical

Move /healthz to a dedicated HTTP server on its own goroutine, independent of the request-forwarding server. This ensures the health endpoint remains responsive even under heavy proxy load. 

// Current: single server handles both health and proxy traffic
// reverse_proxy.go

// Proposed: separate health server on :8445, proxy on :8446
func main() {
    ctx, cancel := signal.NotifyContext(context.Background(),
        os.Interrupt, syscall.SIGTERM)
    defer cancel()

    // Health server — lightweight, isolated from proxy load
    healthMux := http.NewServeMux()
    healthMux.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
        w.WriteHeader(http.StatusOK)
        w.Write([]byte("ok"))
    })
    healthServer := &http.Server{Addr: ":8445", Handler: healthMux}
    go healthServer.ListenAndServe()

    // Proxy server — handles /loki/api/v1/push forwarding
    proxyServer := &http.Server{
        Addr:         ":8446",
        Handler:      proxyHandler,
        ReadTimeout:  30 * time.Second,
        WriteTimeout: 60 * time.Second,
        IdleTimeout:  120 * time.Second,
    }
    go proxyServer.ListenAndServe()

    <-ctx.Done()
    // Graceful shutdown of both servers...
}

2. Add connection/goroutine limits to the proxy Short-term

Prevent unbounded goroutine accumulation when the upstream is slow:

  • Set MaxIdleConnsPerHost on the reverse proxy's http.Transport
  • Add ReadTimeout, WriteTimeout, and IdleTimeout to the proxy server
  • Consider a semaphore or golang.org/x/sync/semaphore to cap concurrent forwarding goroutines
transport := &http.Transport{
    MaxIdleConns:        100,
    MaxIdleConnsPerHost: 50,
    IdleConnTimeout:     90 * time.Second,
    ResponseHeaderTimeout: 30 * time.Second,
}

3. Tune liveness probe thresholds Medium-term

The current probe appears to have tight timeouts. Consider adjusting: 

livenessProbe:
  httpGet:
    path: /healthz
    port: 8445
  initialDelaySeconds: 15
  periodSeconds: 10
  timeoutSeconds: 5        # increase from default 1s
  failureThreshold: 6      # keep at 6 (current)
  successThreshold: 1
Note: Tuning the probe is a mitigation, not a fix. If /healthz is on a separate listener (fix #1), the probe timeout becomes less critical.

4. Export K8s events to Loki for GCPlane Observability

K8s events for the roast namespace are not in Loki, forcing manual CloudWatch queries. Adding event export would:

  • Make liveness/readiness probe failures immediately visible in Grafana
  • Enable alerting on repeated probe failures before pod kill
  • Reduce investigation time from hours to minutes

Siv noted this gap: "want to get these into Loki to make them much easier than this"
